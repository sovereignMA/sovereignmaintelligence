1. Purpose and Scope

This Business Continuity Plan ("BCP") defines the procedures for maintaining and restoring Platform services during and after a disruptive event. It addresses both technical recovery and operational continuity.

The BCP covers all scenarios that could prevent users from accessing Project Sovereign, including:

  • Provider outages (Vercel, Supabase, Stripe, Anthropic)
  • Data loss or corruption events
  • Security incidents requiring service shutdown (refer also to SOV-IRP-001)
  • Loss or incapacitation of the sole operator (Howard Henry)
  • Domain / DNS disruption
  • Credential loss or account lockout

4. Recovery Procedures

4.1 Vercel Outage or Deployment Failure

Detection
Platform returns 5xx or is unreachable
Check Vercel status page: vercel.com/status. Check runtime logs in Vercel Dashboard. If provider outage: monitor Vercel status and wait — no action needed, SLA-backed recovery. If deployment issue:
  1. Check latest deployment state: vercel ls
  2. Roll back to last READY deployment: vercel rollback [deployment-id]
  3. If rollback fails: redeploy from local: vercel deploy --prod
  4. If Vercel account inaccessible: log in via vercel.com with Google auth (trainingcircuitvc@gmail.com)
  5. Communicate outage to affected users if >30 minutes: status update via email

4.2 Supabase Database Outage or Data Loss

For provider outage
Supabase-side incident
Check status.supabase.com. If provider outage: monitor and wait — Supabase manages recovery. The Platform will return HTTP 5xx during the outage. Vercel Edge Middleware will still serve redirects; only DB-dependent routes are affected.
For data corruption or loss
Point-in-Time Recovery
Supabase PITR is enabled on the production project. To restore:

1. Log in to app.supabase.com → project → Settings → Backups
2. Select the recovery point (RPO: any point in the last 7 days)
3. Initiate PITR restore to a new project (non-destructive — creates a new instance)
4. Verify data integrity on the restored project
5. Update SUPABASE_URL and SUPABASE_SERVICE_ROLE_KEY in Vercel env vars to point to restored project
6. Redeploy: vercel deploy --prod
7. Update Supabase edge function secrets: supabase secrets set ...
8. Verify auth, RLS, and all functions operational

4.3 Stripe Account Disruption

  1. Log in to dashboard.stripe.com (trainingcircuitvc@gmail.com + 2FA)
  2. If credentials lost: initiate Stripe account recovery via email
  3. For billing disputes: use Stripe Dashboard to pause subscriptions and communicate with affected users
  4. Subscriptions are stored both in Stripe and mirrored to user_profiles.subscription_status — existing subscribers retain access even if Stripe webhook is temporarily down
  5. New subscriptions: create manually via Stripe Dashboard if checkout API is unavailable

4.4 Loss of Primary Device (HW-001)

  1. All Platform credentials are stored in 1Password (cloud-synced, accessible from any device)
  2. Source code: full clone available from GitHub (sovereignMA/sovereignmaintelligence) — no local-only changes should exist
  3. Acquire replacement device or use any available Mac/PC
  4. Install Vercel CLI: npm i -g vercel; authenticate: vercel login
  5. Clone repo: git clone https://github.com/sovereignMA/sovereignmaintelligence.git
  6. Pull env vars: vercel env pull
  7. Platform administration continues — no code changes needed for operations

4.5 Domain / DNS Disruption

  1. Verify domain registration at registrar (check 1Password for registrar login)
  2. If DNS propagation issue: verify Vercel DNS records in Vercel Dashboard → Domains
  3. Emergency access: direct Vercel deployment URL (sovereign-*.vercel.app) remains accessible
  4. Share emergency URL with known subscribers if domain is down >1 hour
  5. DNS TTL for sovereigncmd.xyz is set to 300 seconds — changes propagate within 5 minutes

4.6 Credential Loss (API Key / Service Account)

  1. All active credentials are listed in SOV-ASR-001 (Section 6)
  2. For each lost credential: access the provider dashboard (1Password has all logins) and generate a new key
  3. Update in Vercel: vercel env add KEY_NAME production
  4. Redeploy: vercel deploy --prod
  5. Update Supabase secrets if applicable: supabase secrets set KEY=value
  6. Update rotation date in SOV-ASR-001

4.7 Anthropic API / AI Outage

  1. Check status.anthropic.com
  2. The Platform degrades gracefully — all non-AI features (pipeline, vault, analytics, email) remain available
  3. AI agents return a user-facing error message; no data is lost
  4. If outage is extended (>4 hours): notify active users via Resend broadcast
  5. No recovery action needed — service resumes automatically when Anthropic API recovers

7. Single-Operator Contingency

Project Sovereign is currently operated by a single person (Howard Henry). The following measures reduce single-operator risk:

  • Credential escrow: All service credentials stored in 1Password with emergency access kit (printed, in secure physical location). Account recovery contacts registered with each provider.
  • Runbook: This BCP serves as the operational runbook for any authorised substitute operator. All recovery procedures are documented without assumed knowledge.
  • Infrastructure as code: All configuration is in vercel.json, middleware.js, and Supabase migrations. A competent developer can reproduce the infrastructure from the GitHub repo alone.
  • Provider accounts: All provider accounts (Vercel, Supabase, Stripe, GitHub) use the trainingcircuitvc@gmail.com Google account as the single sign-on root. Access to Google account grants recovery path for all providers.
  • Subscriber notification: In the event of extended unavailability (>72 hours), affected subscribers will be contacted and offered pro-rata refunds. All user data will be retained and provided on request.
