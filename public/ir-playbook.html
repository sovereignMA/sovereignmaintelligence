1. Purpose and Scope

This Incident Response Playbook ("Playbook") defines the procedures for identifying, containing, eradicating, and recovering from security incidents affecting Project Sovereign. It is the authoritative reference for all security incident response activity.

This Playbook covers:

  • All security events affecting Platform infrastructure, data, or services
  • Personal data breaches under UK GDPR / DPA 2018
  • Service disruption events meeting the P1/P2 severity threshold
  • Third-party provider incidents that affect Platform data or availability

The Incident Response Lead for all incidents is Howard Henry. All incidents must be logged in the compliance_log table (event type: security_incident) within 2 hours of detection.

3. Response Procedures

3.1 All Incidents — Universal Steps

Step 1 — Detect & Triage
Identify and classify the incident
Determine: What happened? When? What systems/data are affected? Assign severity (P1–P4) using Section 2. If uncertain, escalate to P2.

Sources: Vercel runtime logs · Supabase logs · admin audit trail · health-check cron alerts · user reports · Stripe webhook failures
Step 2 — Log & Preserve
Create an incident record immediately
Log to compliance_log table: event_type = security_incident, framework = severity level, notes = brief description, timestamp.
Preserve evidence: export relevant Vercel logs, Supabase audit_trail rows, and Stripe event logs. Do not delete or modify anything until investigation is complete.
Step 3 — Contain
Stop the bleeding — limit blast radius
Immediate containment actions (as appropriate): revoke compromised credentials · disable affected user accounts via Admin Users panel · put Vercel deployment on maintenance mode · pause cron jobs · block suspicious IPs via Vercel firewall rules.
Step 4 — Eradicate
Remove the root cause
Rotate all potentially compromised secrets. Patch vulnerabilities. Remove unauthorised access. Verify no backdoors or persistent access remain. Cross-check asset register (SOV-ASR-001) to confirm all affected assets identified.
Step 5 — Recover
Restore to secure operations
Restore affected services using BCP procedures (SOV-BCP-001). Verify integrity of restored data. Monitor for 24 hours post-recovery for recurrence. Update compliance_log with resolution timestamp and root cause.
Step 6 — Post-Incident Review
Document and improve
Complete a post-incident review within 5 business days: root cause analysis, timeline, impact assessment, remediation actions completed, and preventive measures. Update this Playbook if procedure gaps were identified.

3.2 Personal Data Breach — UK GDPR Protocol

If the incident involves personal data (any data relating to an identified or identifiable natural person), the following additional obligations apply:

T+0 — Discovery
Clock starts now
The 72-hour ICO notification window begins at the moment personal data breach is known (not suspected). Assess: What data? How many individuals? What risk of harm?
T+24 hours
Initial risk assessment complete
Determine whether breach is notifiable to ICO (Art. 33 UK GDPR): breach is notifiable unless it is unlikely to result in a risk to rights and freedoms of individuals. If notifiable, begin preparing ICO notification. If clearly not notifiable, document the reasoning.
T+72 hours
ICO notification deadline (if notifiable)
Report via ICO self-reporting portal: ico.org.uk/for-organisations/report-a-breach
Required information: nature of breach · categories and approximate number of individuals · categories and approximate number of records · DPO contact details · likely consequences · measures taken/proposed.
If full information is not available within 72 hours, submit initial notification and follow up without undue delay. Partial notification is better than no notification.
Without undue delay
Data subject notification (if high risk)
If the breach is likely to result in high risk to affected individuals (Art. 34 UK GDPR), notify affected users directly via email. Message must: describe the breach in plain language · provide DPO contact details · describe likely consequences · describe measures being taken. Draft and send via Resend.

3.3 Credential / Secret Compromise

  1. Immediately revoke the compromised credential in the issuing service (Stripe Dashboard, Supabase Settings, Anthropic Console, etc.)
  2. Generate a new credential and update in Vercel environment variables: vercel env add KEY production
  3. Redeploy: vercel deploy --prod
  4. Audit audit_trail for any activity using the compromised credential within the suspected exposure window
  5. Update asset register (SOV-ASR-001) with new rotation date
  6. If Supabase service role key: immediately check all tables for unauthorised data changes; restore from PITR if necessary

3.4 Unauthorised Account Access

  1. In Admin Users panel: disable the affected account immediately
  2. Revoke all active sessions via Supabase Auth dashboard
  3. Export and review audit_trail for all actions taken by the account during the suspected window
  4. If personal data was accessed: trigger Personal Data Breach protocol (3.2)
  5. Contact affected user to notify, change credentials, and re-enable if legitimate user
  6. Review how unauthorised access was obtained; patch the vector before re-enabling

3.5 Stripe / Payment Incident

  1. If fraudulent charges suspected: contact Stripe Trust & Safety immediately via Stripe Dashboard
  2. Revoke STRIPE_SECRET_KEY and generate a new key; redeploy
  3. Review all Stripe webhook events in the past 30 days via Stripe Dashboard Events log
  4. Check user_profiles for unexpected subscription status changes
  5. If card data exposure: note that Platform uses Stripe-hosted Checkout — no raw card data ever touches Platform servers. Stripe handles PCI DSS compliance. Incident is Stripe-side; follow their notification.

3.6 AI Prompt Injection / Data Exfiltration via AI

  1. Disable AI agents immediately: set ANTHROPIC_API_KEY to empty string in Vercel env vars; redeploy
  2. Review conversations table for the affected user/session to understand what data was exposed in prompts
  3. Review audit_trail for the ai-proxy function calls during the incident window
  4. If personal data was extracted: trigger Personal Data Breach protocol (3.2)
  5. Re-enable AI only after root cause is identified and input sanitisation is verified

5. Evidence Preservation Checklist

For all P1/P2 incidents, preserve the following before taking remediation actions where possible:

  • Vercel runtime logs export (from Vercel Dashboard → Deployments → Logs)
  • Supabase audit_trail rows for affected user and time window (SQL export)
  • Supabase auth logs (Supabase Dashboard → Auth → Logs)
  • Stripe event log export for relevant time window
  • Browser/device access logs if client-side compromise suspected
  • Screenshot or export of any error messages or unusual UI state
  • Timestamps in UTC for all observed anomalies

Evidence must be retained for a minimum of 3 years from incident date, or as required by any legal/regulatory proceedings.

7. Testing and Review

This Playbook is tested and reviewed as follows:

  • Annual tabletop exercise: Simulate a P1 personal data breach scenario; walk through all steps; identify gaps
  • Annual review: Review all procedures against current infrastructure and any incidents from the past year
  • Post-incident update: Update Playbook within 5 business days of any P1/P2 incident to address identified gaps
  • Regulation tracking: Review ICO guidance and NCSC advisories quarterly; update if material changes occur
